# Scheduled Task SOP

Directories: `../sche_tasks/` holds task definitions (JSON), `../sche_tasks/done/` holds execution reports.

## Task JSON format (*.json)
```json
{"schedule":"08:00", "repeat":"daily", "enabled":true, "prompt":"...", "max_delay_hours":6}
```
`repeat` options: daily | weekday | weekly | monthly | once | every_Nh (every N hours) | every_Nd (every N days).
`max_delay_hours` (optional, default 6): if more than this many hours past `schedule` have passed, the task no longer fires — prevents stale tasks from firing after a late boot.

## Trigger flow
1. `scheduler.py` (in reflect/) polls `sche_tasks/*.json` every 60 seconds.
2. The task fires only when all conditions are met: `enabled=true` + current time >= schedule + cooldown elapsed (based on the latest report timestamp under `done/`).
3. On trigger, the prompt is assembled with the report path: `../sche_tasks/done/YYYY-MM-DD_taskname.md`.
4. **First thing after receiving a task**: use update_working_checkpoint to record the report target path, so a long-running task does not forget it.
5. After execution, write the report to that path (the scheduler uses the file's existence to decide that today is done).

## Logs and monitoring
- The scheduler auto-logs to `sche_tasks/scheduler.log` (triggered/skipped/error).
- `scheduler.health_check()` returns the status list of every task (HEALTHY/OVERDUE/DISABLED/NEVER_RUN/ERROR).
- JSON parse errors, schedule format errors and unknown repeat types are all logged.

## Notes
- `once`: cools down for 100 years after a single execution (effectively permanent skip).
- The task file only describes "what to do"; the report path is auto-generated by the scheduler and injected into the prompt.
- The `sche_tasks` directory lives in `../`, i.e. the code root.
